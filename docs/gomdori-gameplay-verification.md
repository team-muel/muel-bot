## 1. 검증 3층 — 무엇을 어디서 재는가

| 층 | 도구 | 답하는 질문 |
|---|---|---|
| 정합 | `npm run test:gomdori` (7 스위트) | 능력·판정이 명세대로 동작하는가 |
| **구조** | **`npm run sim:balance` (본 문서)** | 시스템 자체가 어느 쪽으로 기울었나, degenerate 가 있나 |
| 경험 | 완성본 → 경험자 피드백 + M9 라이브 지표 | 재미있는가, 페이스가 맞는가 (잠금 #6) |

구조 시뮬의 정책 = **uninformed baseline**: 투표·능력 대상은 합법 범위 내 균등 랜덤,
악마팀만 서클을 알고 회피(실전과 같은 정보 비대칭). 추리 정보가 0일 때 시스템의 기울기를
재는 것이므로 **실플레이 승률이 아니다** — 추리는 천사 쪽 가산이므로 실전 천사 승률은
baseline 보다 항상 높다. 반대로, baseline 에서도 보이는 신호는 추리로도 못 덮는
**구조의 문제**일 확률이 높다.

## 3. 발견 — 우선순위순

### P0-A. 중립 파이프라인 단절 — 파스아는 라이브에서 등장 자체가 불가능
- muel-tree 로비의 중립 토글은 `match-settings` edge function 을 호출하는데, **이 함수는
  레포에도 라이브(15개 함수 목록)에도 존재하지 않는다** → 호스트가 누르면 404 "설정 변경 실패".
- `match-start` 는 `settings.includeNeutral === true` 만 읽고, `settings.neutral`("auto"/"on"/"off")
  은 서버 어디서도 해석하지 않는다. 어떤 경로로도 `includeNeutral` 이 true 가 되지 않으므로
  **현재 라이브에서 파스아 등장 확률은 0**. 로비 카피("자동: 나올지 아무도 모릅니다")는
  엔진과 불일치 — M3-1 의 UI 반쪽만 배포된 상태(결정 잠금 #2 미반영).
- 수리 범위: ① `match-settings` 함수 신설(host 검증 + settings merge) ② `match-start` 가
  `resolveNeutralMode` 동일 규칙으로 auto=확률(수치는 잠금 #5 게이트에서) / on / off 해석.

### P0-B. 부활 무한 루프 = 교착 엔진 (무한게임 degenerate 실증)
- 7인+ 교착 14.8~37.0%, 교착 판은 40일 캡 도달 — 예상 실시간 2~3시간(Discord 세션 불가).
  부활/판이 12인 14.3회까지 치솟는 것이 원인: **미즐렛·헬렌 부활에 횟수 제한이 없어**
  밤마다 사망 ↔ 부활이 상쇄되어 게임이 수렴하지 않는다.
- `maxUses` 가 타입에 선언만 있고 엔진/match-action 어디서도 강제되지 않는 것의 실해(實害).
  (팬텀 일식은 시전자 자기소멸이라 자연 1회 — 무해.)
- 처방 후보(택1+): 부활 `maxUses: 1` + **엔진 차원의 maxUses 강제 구현**(범용) /
  같은 대상 재부활 금지 / 부활 시 시전자 소모(1회성 소명). canon 원문 확인 후 결정.
- 교착의 보조 요인으로 **최대 일수 강제 종료(M2-5)도 미구현** — 부활을 묶어도 안전망은 필요.

### P0-C. 파스아 지배 — "밸런스 조절자"가 아니라 시계
- 등장 시 중립승 41.4%(8인) → **87.0%(12인)**, 인원이 늘수록 단조 증가. 임계가 인원 무관
  **고정 3** 이라 큰 판일수록 먹이만 늘고 시계는 같다. 평균 6일 — 파스아가 사실상 게임 길이를
  결정하고, 악마승도 6~48%로 압축(존재만으로 양 진영 승리를 흡수).
- baseline 한계 주의: 전향자가 자기 전향을 아는 사회적 신호(폭로·암시)는 시뮬에 없다 —
  실전 중립승은 이보다 낮다. 그래도 인원↑=중립승↑ 의 **단조 구조**는 정책과 무관한 신호.
- 처방 후보: 임계 인원 비례(예: `ceil(N/3)`) / 전향 쿨다운(격밤) / 전향자 사망 시 교세 차감.
  수치는 잠금 #5 게이트 — 이 시뮬로 후보안 비교 측정 후 결정.

### P1-D. 5인 구성 과중 — 악마팀 고정 2가 최소 인원을 부순다
- 5인 악마승 95.8%, 평균 2.9일, 처형 0.5회 — 사실상 시작 전에 끝나 있다. 2(악마+조력자) vs 3
  에서 밤 1킬이면 2:2 패리티. 추리가 끼어들 시간 자체가 없다(p90 = 3일).
- 처방 후보: 5~6인은 조력자 제외(악마 1만) / 5인 한정 첫 밤 처치 불가 유지+악마승 조건을
  "초과"로(>= → >) / 최소 인원 6으로 상향. 구성 변경이라 잠금 #1(전체 로스터)과 무관 —
  인원별 분포 정책의 문제.

### P1-E. 위생 — maxUses 규약화
- 타입 선언만 있는 `maxUses` 를 엔진이 읽도록 구현하고(P0-B 와 동일 작업), 신규 능력 추가
  스킬/체크리스트에 "횟수 제한은 maxUses 로만"을 명시해 하드코딩 변종을 차단.

## 6. 후속 ALL — 인원 정책 8~12 + 파스아 임계 채택 (2026-06-11)

**인원 정책**: 사용자 확정 — "5인 게임은 의도한 결과 아님, 원본은 8~12".
`gomdori-rules.playerCount {min:8, max:12}` 단일 출처로 match-start 검증·로비 시작
조건·시뮬 범위 전부 이행. §2/§5 의 5~7인 행은 역사적 참고(P1-D 는 정책으로 해소).
부수 효과: 모든 유효 게임이 중립 등장 자격(8인+)을 가진다.

**파스아 임계 후보 비교** (`sim:balance --rule`, N=1500, seed 42, 등장 게임 중립승 %):

| fixed3 (구) | 35.5 | 50.5 | 59.5 | 68.1 | 70.1 | 인원↑=중립↑ 단조, 지배 |

**채택: scale-alive** — `engine.pasuaWinThreshold(N) = max(3, ceil(N/3))` (8~9인 3,
10~12인 4) + 교세는 **생존 전향자만**(countTeams). 선정 근거: ① 등장 게임 중립승
16~34%로 유일하게 "조절자" 범위 ② 인원 단조성 파괴(임계 점프) ③ **전향자 처형 =
교세 차감**이라는 카운터플레이 서사가 생긴다 — "누가 전향됐는가"가 사회적 추리의
대상이 됨. 잠금 #5 의 수치 결정을 "후속 ALL" 지시로 집행, 되돌리기는 함수 한 곳.

**auto 등장 확률 = 0.5 유지** (#123 의 `NEUTRAL_SPAWN_CHANCE`): 임계 수리로 파스아
게임이 더는 degenerate 가 아니고, 50% 가 "존재를 알 수 없다"(잠금 #2)의 불확실성을
최대화한다. 라이브 지표에서 중립 등장 게임의 완주율·만족이 처지면 재조정.
